## 6. Recommendations

### When graph navigation excels ⭐

**Multi-file refactoring** (73.7% avg reduction):
- Module rename/move operations
- Import updates
- Code extraction/consolidation
- Dependency graph provides complete import map instantly

**Circular dependency detection** (91.7% avg reduction):
- Identify import cycles without reading files
- Fix cycles by reading only involved files
- Graph traversal detects cycles in O(n) time vs O(n²) for manual inspection

**Dependency analysis** (57.4% avg reduction):
- Find callers/callees of functions
- Trace call chains
- Impact analysis
- Graph provides exact dependency relationships

**Large codebases** (>20 files):
- More files = more grep false positives
- Graph navigation scales better (constant overhead vs linear search)

### When graph navigation is neutral

**Single-file tasks**:
- If task only needs one file, graph overhead not beneficial
- Example: "What does function X do in file Y?"
- Recommendation: Use direct Read tool

**Small codebases** (<10 files):
- Grep is fast enough
- Graph overhead may not justify savings
- Exception: If circular dependencies are concern

### When graph navigation might fail

**Dynamic imports** (not captured in static graph):
- Runtime imports via `__import__()`
- Plugin systems with dynamic loading
- Workaround: Hybrid approach (graph + runtime analysis)

**Cross-language dependencies** (if not supported):
- Python calling JavaScript
- Mixed-language codebases
- Workaround: Language-specific graphs or manual tracing

**Complex control flow** (reflection, eval, etc.):
- Dynamically constructed function calls
- Code generation
- Workaround: Combine graph with runtime profiling

**Current limitations**:
- Ground truth incomplete for some test tasks
- Non-Python files not fully supported (metadata only)
- No AST-level graph visualization yet
